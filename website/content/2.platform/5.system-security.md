## Overview

Kinotic OS implements a three-layer Identity and Access Management (IAM) system that separates authentication across distinct scopes. Each layer maintains its own user pool with complete isolation — users in one scope cannot access another.

## Three-Layer IAM Model

| Layer | Scope | Purpose |
|-------|-------|---------|
| **System** | Platform-wide | Platform operators managing the Kinotic OS deployment |
| **Organization** | Per-organization | Development teams building applications on the platform |
| **Application** | Per-application | End-users and machines consuming deployed applications |

### Scope Isolation

Each `IamUser` is scoped to exactly one layer via `authScopeType` and `authScopeId`. A user with email `jane@example.com` scoped to Organization A is a completely separate identity from `jane@example.com` scoped to Application B. There is no cross-scope access.

### Authentication Methods

Both methods are available at every scope layer:

- **Email/Password (LOCAL)** — User credentials are stored as bcrypt hashes in a separate `IamCredential` entity. Password hashes are never exposed through user CRUD operations.
- **OIDC (Federated)** — JWT tokens are validated against OIDC configurations enabled for the scope. The platform verifies the token's signature, issuer, audience, and expiration.

## IamUser Entity

The `IamUser` entity represents an authenticated identity at any scope layer.

| Field | Description |
|-------|-------------|
| `id` | UUID |
| `email` | Unique within scope |
| `displayName` | Human-readable name |
| `authType` | `LOCAL` or `OIDC` |
| `oidcSubject` | Subject claim from OIDC token (populated on first OIDC login) |
| `oidcConfigId` | Reference to the OidcConfiguration used for authentication |
| `authScopeType` | `SYSTEM`, `ORGANIZATION`, or `APPLICATION` |
| `authScopeId` | `null` for system, org ID for organization, app ID for application |
| `enabled` | Whether the user can authenticate |

### User Pre-Creation Requirement

Users must be pre-created by an administrator before they can authenticate. This ensures that a valid Google or Microsoft account does not automatically grant access to any scope. For OIDC users, the administrator creates the user with their email — the `oidcSubject` is populated automatically on first successful login.

## OIDC Configuration

The `OidcConfiguration` entity is a standalone, reusable OIDC provider configuration. It has no embedded scope or ownership — the association between OIDC configs and the scopes that use them is stored on the consuming entity.

| Field | Description |
|-------|-------------|
| `id` | UUID |
| `name` | Human-readable name (e.g., "Google", "Microsoft") |
| `provider` | Provider identifier (e.g., "google", "okta", "azure-ad") |
| `builtIn` | `true` for platform-provided configs — immutable for org/app admins |
| `clientId` | OAuth client ID |
| `authority` | Issuer URL — must match the `iss` claim in JWTs |
| `backchannelAuthority` | Cluster-internal URL for token validation (optional) |
| `redirectUri` | OAuth redirect URI |
| `domains` | Email domains this provider handles (used for matching) |
| `audience` | Expected `aud` claim in JWTs |
| `rolesClaimPath` | Dot-separated path to roles in JWT claims (e.g., `realm_access.roles`) |
| `enabled` | Whether this configuration is active |

### Built-In Configurations

Configs marked `builtIn=true` are platform-provided — they represent Kinotic OS's own registration with identity providers like Google and Microsoft. Built-in configs are:

- Created by system administrators
- Visible to all organization and application administrators
- Immutable for non-system users (read-only + enable)
- Shared across all scopes that reference them

To enable a built-in config for a scope, add its ID to the entity's `oidcConfigurationIds` list.

### Enabling OIDC for a Scope

Each scope layer stores a list of `oidcConfigurationIds` on its entity:

| Entity | Scope Layer | Service |
|--------|-------------|---------|
| `KinoticSystem` | System | `KinoticSystemService` |
| `Organization` | Organization | `OrganizationService` |
| `Application` | Application | `ApplicationService` |

To enable an OIDC provider for a scope, add the `OidcConfiguration` ID to the entity's list and save it. Multiple scopes can reference the same configuration ID — no duplication needed.

## Authentication Flows

### Email/Password Flow

1. Client connects with `login`, `passcode`, `authScopeType`, and `authScopeId` headers
2. Platform looks up `IamUser` by email within the specified scope
3. Retrieves the `IamCredential` (password hash) for that user
4. Verifies the bcrypt hash
5. Returns a session with the user's identity and scope as the tenant context

### OIDC Flow

1. Client connects with `Authorization: Bearer <jwt>`, `authScopeType`, and `authScopeId` headers
2. Platform loads the OIDC configurations enabled for the specified scope
3. Parses the JWT to extract the issuer and email domain
4. Matches against the available configurations (issuer matches authority, email domain matches domains list)
5. Validates the JWT signature, audience, and expiration using the matched configuration
6. Looks up the `IamUser` by email within the scope — rejects if no pre-created user exists
7. Populates `oidcSubject` on first login
8. Returns a session with the user's identity and scope as the tenant context

## KinoticSystem Entity

The `KinoticSystem` entity is a singleton representing the Kinotic OS deployment itself. It holds the `oidcConfigurationIds` for system-level authentication.

| Field | Description |
|-------|-------------|
| `id` | Fixed value: `kinotic-system` |
| `oidcConfigurationIds` | OIDC configs available for system-level auth |
| `updated` | Last modification timestamp |

Managed via `KinoticSystemService.getSystem()` and `KinoticSystemService.save()`.

## Bootstrap

New deployments are bootstrapped via the migration process. The `V3__iam.sql` migration creates all IAM tables and can seed:

- An initial `KinoticSystem` singleton record
- An initial system administrator `IamUser` and `IamCredential` with a known bootstrap password

No startup-time bootstrap logic is needed — migrations always run before initial boot.

## Network Security

- All client-to-server communication occurs over WebSocket with TLS
- Internal service-to-service communication is secured within the Kubernetes cluster network
- TLS termination is handled at the ingress layer with support for automated certificate management

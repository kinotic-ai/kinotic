## Overview

Kinotic OS implements a three-layer Identity and Access Management (IAM) system that separates authentication across distinct scopes. Each layer maintains its own user pool with complete isolation — users in one scope cannot access another.

## Three-Layer IAM Model

| Layer | Purpose |
|-------|---------|
| **System** | Platform operators managing the Kinotic OS deployment |
| **Organization** | Development teams building applications on the platform |
| **Application** | End-users and machines consuming deployed applications |

A user with email `jane@example.com` in Organization A is a completely separate identity from `jane@example.com` in Application B. There is no cross-scope access, even if both scopes use the same OIDC provider.

### Authentication Methods

Both methods are available at every scope layer:

- **Email/Password** — Credentials are stored with bcrypt hashing. Password hashes are kept in a separate internal entity and never exposed through user management APIs.
- **OIDC (Federated)** — JWT tokens are validated against OIDC configurations enabled for the scope. The platform verifies the token's signature, issuer, audience, and expiration.

### User Pre-Creation

Users must be pre-created by an administrator before they can authenticate. This is a deliberate design — having a valid Google account does not automatically grant access to any scope. For OIDC users, the administrator only needs the user's email. The OIDC identity link is established automatically on first successful login.

## OIDC Configuration

OIDC configurations are standalone, reusable objects with no embedded scope or ownership. The association between a config and the scopes that use it is stored on the consuming entity (System, Organization, or Application) as a list of configuration IDs.

### Built-In Configurations

System administrators can create **built-in** OIDC configurations that represent Kinotic OS's own registration with providers like Google and Microsoft. Built-in configs are:

- Visible to all organization and application administrators
- Immutable for non-system users (read-only — they can enable but not modify)
- Shared across all scopes that reference them

This means Kinotic OS registers once with Google, and every customer's organization and application can enable Google login without touching the provider's console. End users see "Kinotic OS" on the consent screen.

### Enabling OIDC for a Scope

To enable an OIDC provider, add the configuration's ID to the scope entity's `oidcConfigurationIds` list. Multiple scopes can reference the same configuration — no duplication needed. The same config can serve system login, org login, and app login simultaneously.

## Authentication Flows

### Email/Password

1. Client connects with `login`, `passcode`, `authScopeType`, and `authScopeId` headers
2. Platform looks up the user by email within the specified scope
3. Verifies the password
4. Returns a session with the user's identity and scope as the tenant context

### OIDC

1. Client connects with `Authorization: Bearer <jwt>`, `authScopeType`, and `authScopeId` headers
2. Platform loads the OIDC configurations enabled for the scope
3. Matches the JWT's issuer and email domain against the available configurations
4. Validates the JWT signature, audience, and expiration
5. Looks up the user by email within the scope — rejects if no pre-created user exists
6. Returns a session with the user's identity and scope as the tenant context

## Bootstrap

New deployments are bootstrapped via the migration process. The IAM migration creates all necessary tables and seeds the initial system administrator account. No startup-time bootstrap logic is needed — migrations always run before initial boot.

## Network Security

- All client-to-server communication occurs over WebSocket with TLS
- Internal service-to-service communication is secured within the Kubernetes cluster network
- TLS termination is handled at the ingress layer with support for automated certificate management

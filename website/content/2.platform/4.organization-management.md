## Overview

Kinotic OS uses an organization model to manage access and resources across the platform. Organizations provide the boundary for teams, applications, users, and shared OIDC configuration.

## Organizations

An organization represents a development team or company building applications on the platform. Each organization gets a URL-safe slug auto-generated from its name and tracks which administrator created it.

Organizations are managed through the `OrganizationService`, which provides standard CRUD operations.

## User Management

Users within an organization are created and managed through the `IamUserService`. Each user is scoped to the organization — they exist only within that organization's user pool.

### Local Users

Create users with email/password authentication by calling `createUser` with a password. The password is bcrypt-hashed and stored separately from the user record — it is never exposed through any API.

### OIDC Users

Create users who will authenticate via a federated provider (Google, Microsoft, etc.) by calling `createUser` with no password. The administrator only needs the user's email address. On first OIDC login, the platform matches the JWT email to the pre-created user and links the OIDC identity automatically.

### Password Management

Administrators can reset a user's password directly. Users can change their own password by providing their current password.

## OIDC Configuration

Organizations can enable OIDC providers for their users by referencing OIDC configuration IDs.

### Enabling a Built-In Provider

Built-in OIDC configurations are pre-registered by system administrators (e.g., Google, Microsoft). Organization administrators can browse available built-in configs and enable them by adding the config ID to the organization. Built-in configurations are read-only for organization administrators.

### Creating a Custom Provider

Organization administrators can also create custom OIDC configurations for providers not covered by built-in configs (e.g., a corporate Okta instance). Create the configuration, then add its ID to the organization.

### Sharing with Applications

The same OIDC configuration can be enabled for both an organization and its applications. Add the same config ID to the application's configuration — no duplication needed, both reference the same underlying config.

## Application Users

Application-level users are managed the same way as organization users but in a separate user pool. A user in an organization and a user in one of its applications are completely independent identities, even if they share the same email address. This allows developers to use email/password for org-level access while testing, and federated SSO for their application's end users.

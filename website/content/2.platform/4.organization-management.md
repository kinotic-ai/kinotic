## Overview

Kinotic OS uses an organization model to manage access and resources across the platform. Organizations provide the boundary for teams, applications, and shared configuration.

## Organization Entity

An organization represents a development team or company building applications on the platform.

| Field | Description |
|-------|-------------|
| `id` | UUID |
| `name` | Organization name |
| `slug` | URL-safe identifier derived from the name (auto-generated) |
| `description` | Human-readable description |
| `oidcConfigurationIds` | OIDC configurations enabled for this organization |
| `createdBy` | User ID of the creator |
| `created` | Creation timestamp |
| `updated` | Last modification timestamp |

Organizations are managed through the `OrganizationService`, which provides standard CRUD operations. The `slug` is automatically generated from the `name` when an organization is created or updated.

## User Management

Users within an organization are managed through the `IamUserService`. Each user is scoped to the organization via `authScopeType=ORGANIZATION` and `authScopeId=<orgId>`.

### Creating a Local User

Create users with email/password authentication:

```json
{
    "email": "developer@example.com",
    "displayName": "Jane Developer",
    "authType": "LOCAL",
    "authScopeType": "ORGANIZATION",
    "authScopeId": "<organization-id>"
}
```

Call `IamUserService.createUser(user, password)` — the password is bcrypt-hashed and stored in a separate credential entity.

### Creating an OIDC User

Create users who will authenticate via an OIDC provider (Google, Microsoft, etc.):

```json
{
    "email": "developer@example.com",
    "displayName": "Jane Developer",
    "authType": "OIDC",
    "authScopeType": "ORGANIZATION",
    "authScopeId": "<organization-id>"
}
```

Call `IamUserService.createUser(user, null)` with no password. On first OIDC login, the platform matches the JWT email to this user and automatically populates the `oidcSubject` and `oidcConfigId` fields.

### Password Management

- `IamUserService.changePassword(userId, currentPassword, newPassword)` — User-initiated password change (requires current password)
- `IamUserService.resetPassword(userId, newPassword)` — Admin-initiated password reset

### Querying Users

- `IamUserService.findByEmailAndScope(email, authScopeType, authScopeId)` — Look up a specific user
- `IamUserService.findByScope(authScopeType, authScopeId, pageable)` — List all users in a scope

## OIDC Configuration

Organizations can enable OIDC providers for their users by adding OIDC configuration IDs to the organization's `oidcConfigurationIds` list.

### Enabling a Built-In Provider

Built-in OIDC configurations are pre-registered by system administrators (e.g., Google, Microsoft). To enable one for your organization:

1. Query `OidcConfigurationService` for available configurations with `builtIn=true`
2. Add the desired configuration ID to your organization's `oidcConfigurationIds`
3. Save the organization via `OrganizationService.save()`

Built-in configurations are read-only for organization administrators — they cannot be modified or deleted.

### Creating a Custom Provider

Organization administrators can also create custom OIDC configurations for providers not covered by built-in configs:

```json
{
    "name": "Corporate Okta",
    "provider": "okta",
    "builtIn": false,
    "clientId": "<okta-client-id>",
    "authority": "https://your-org.okta.com",
    "redirectUri": "https://your-domain/auth/callback",
    "domains": ["your-company.com"],
    "audience": "<okta-client-id>",
    "enabled": true
}
```

Create via `OidcConfigurationService.save()`, then add the returned ID to the organization's `oidcConfigurationIds`.

### Sharing OIDC Configs with Applications

The same OIDC configuration can be enabled for both an organization and its applications. To enable a config for an application, add its ID to the application's `oidcConfigurationIds` list and save via `ApplicationService.save()`. No duplication is needed — both entities reference the same configuration by ID.

## Application User Management

Application-level users are managed the same way as organization users, but with `authScopeType=APPLICATION` and `authScopeId=<appId>`. Application users are completely isolated from organization users — they are separate identity pools.

::alert{type="info"}
**Teams and Roles** — Team management and role-based organization features are under active development. Roles referenced in ABAC policies are currently managed directly on user metadata.
::

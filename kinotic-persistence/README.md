# kinotic-persistence

A Spring Boot library that provides schema-driven entity storage over Elasticsearch. The preferred access path is the native Java service API (`EntitiesService`, `JsonEntitiesService`); the library also auto-generates OpenAPI REST and GraphQL endpoints for external HTTP consumers.

## Overview

Applications that store structured, multi-tenant data typically must wire together a persistence layer, a schema registry, query execution, and externally-facing HTTP endpoints by hand. `kinotic-persistence` eliminates that work by treating an `EntityDefinition` — a named schema enriched with IDL decorators — as the single source of truth. When an `EntityDefinition` is published, the library derives an Elasticsearch index mapping, generates OpenAPI path items, and builds a GraphQL schema segment, all without application code.

The module sits at the data layer of the broader Kinotic architecture. It depends on `kinotic-core` for configuration and security primitives, `kinotic-domain` for shared service contracts and pagination types, `kinotic-idl` for the type system (`ObjectC3Type`, `C3Decorator`, etc.) that backs every entity schema, `kinotic-rpc-gateway` for remote service publication (`@Publish`), and `kinotic-sql` for named-query SQL parsing and execution. Consumers interact with the module through a small set of stable service interfaces; everything else is `internal`.

Multi-tenancy is a first-class concern. Each `EntityDefinition` declares a `MultiTenancyType` (`NONE` or `SHARED`). When `SHARED`, a designated tenant-id field is injected automatically on every write and used to filter every read unless the caller holds a `tenantSelection`. A parallel `AdminJsonEntitiesService` allows privileged callers to supply an explicit tenant list and cross tenant boundaries in a controlled way.

Vert.x is used for all I/O. OpenAPI requests are handled by `OpenApiVerticle`, GraphQL requests by `GqlVerticle`. Both verticles are deployed in parallel, one instance per available core. A periodic health check pings Elasticsearch and reports results through Vert.x Health Checks, which are served on a dedicated web server port.

## Key Concepts

- **EntityDefinition** — The schema record that drives everything. It carries a name, application ID, project ID, an `ObjectC3Type` schema, decorator metadata, and lifecycle flags (`published`, `created`, `updated`). Only published definitions are served through the HTTP endpoints.

- **Decorator** — A `C3Decorator` subclass attached to a type or field inside an `ObjectC3Type`. Decorators control Elasticsearch mapping (e.g., `TextDecorator` → `text` field type, `NestedDecorator` → nested object), identity (`IdDecorator`, `AutoGeneratedIdDecorator`), multi-tenancy (`TenantIdDecorator`), versioning (`VersionDecorator`), and access control (`PolicyDecorator`, `RoleDecorator`). The full set is described in the [Configuration](#configuration) section.

- **EntityContext** — A per-request value object that bundles the calling `Participant` (identity), an optional field inclusion filter, and an optional tenant selection. Every service method receives one and every Vert.x handler constructs one from the authenticated routing context.

- **EntitiesService** — The **preferred** access path for Java consumers. A type-safe CRUD interface that accepts plain Java objects and serializes/deserializes them through Jackson. Prefer this over the HTTP endpoints whenever the caller runs in the same JVM.

- **JsonEntitiesService / AdminJsonEntitiesService** — JSON-level service interfaces published via `@Publish` for remote callers (e.g., the RPC gateway). `JsonEntitiesService` operates within the caller's tenant; `AdminJsonEntitiesService` accepts an explicit `tenantSelection` list and is intended for administrative tooling. Use these when calling from outside the JVM; prefer `EntitiesService` for in-process callers.

- **NamedQueriesService** — Manages `NamedQueriesDefinition` records that associate SQL statements with a `QueryDecorator`-annotated function inside an entity's IDL schema. Named queries are executed against Elasticsearch's SQL endpoint and results are projected back to entity objects.

- **UpsertPreProcessor / DecoratorLogic** — A pipeline that runs before every save or update. Each decorator that carries write-time semantics (auto-ID generation, tenant-id injection, version increment, time reference stamping) is paired with an `UpsertFieldPreProcessor` implementation. The pipeline is built once per `EntityDefinition` and cached.

- **GqlVerticle / OpenApiVerticle** — Vert.x `VerticleBase` implementations that host the HTTP servers. `GqlVerticle` routes POST requests to a `DelegatingGqlHandler` that resolves the application from the URL path parameter (`structureApplication`) and dispatches to a per-application GraphQL schema. `OpenApiVerticle` delegates to a generated Vert.x Web router built from the OpenAPI spec for each application.

| `kinotic.persistence.indexPrefix` | `kinotic_` | Prefix applied to all Elasticsearch index names |

| `kinotic.persistence.elasticUsername` | `null` | Elasticsearch username (optional) |
| `kinotic.persistence.elasticPassword` | `null` | Elasticsearch password (optional) |
| `kinotic.persistence.elasticConnectionTimeout` | `5s` | Connection timeout for the Elasticsearch client |
| `kinotic.persistence.elasticSocketTimeout` | `1m` | Socket timeout for the Elasticsearch client |
| `kinotic.persistence.elasticHealthCheckInterval` | `1m` | Interval between Elasticsearch cluster health polls |
| `kinotic.persistence.openApiPort` | `8080` | Port for the OpenAPI REST server |

### Auto-configuration

`KinoticPersistenceAutoConfiguration` (in package `org.kinotic.persistence_autoconfig`, intentionally outside component-scan) imports `KinoticPersistenceLibrary`. The library class is guarded by `@ConditionalOnProperty(value = "kinotic.disablePersistence", havingValue = "false", matchIfMissing = true)`.

### Decorator reference

Decorators are `C3Decorator` subclasses from `org.kinotic.persistence.api.model.idl.decorators`. They are attached to properties or types inside an `ObjectC3Type` schema before passing that schema to `EntityDefinition.setSchema()`.

#### Entity decorators (target: TYPE)

| Decorator class | IDL type string | Description |
|---|---|---|
| `EntityDecorator` | `Entity` | Marks an `ObjectC3Type` as a persistence entity. Carries `multiTenancyType` (`NONE` or `SHARED`) and `entityType` (`TABLE` or `STREAM`). Required for every entity schema. |
| `EntityServiceDecoratorsDecorator` | `EntityServiceDecorators` | Attaches an `EntityServiceDecoratorsConfig` to a type, mapping CRUD operation groups (`allCreate`, `allRead`, `allUpdate`, `allDelete`) or individual operations to lists of `PolicyDecorator` or `RoleDecorator` instances. |
| `EsIndexConfigurationDecorator` | `EsIndexConfigurationDecorator` | Overrides the default Elasticsearch index configuration. Carries an `EsIndexConfigurationData` with a set of component template names to apply. |

#### ID decorators (target: FIELD)

| Decorator class | IDL type string | Description |
|---|---|---|
| `IdDecorator` | `Id` | Marks a field as the entity ID. The caller must supply this value on every write. |
| `AutoGeneratedIdDecorator` | `AutoGeneratedId` | Marks a field as an auto-generated ID. The server generates a UUID if the field is absent on save. |

#### Field mapping decorators (target: FIELD)

| Decorator class | IDL type string | Description |
|---|---|---|
| `TextDecorator` | `Text` | Maps a `StringC3Type` field as an Elasticsearch `text` field, enabling full-text analysis. |
| `NotIndexedDecorator` | `NotIndexedDecorator` | Excludes the field from Elasticsearch indexing entirely. |
| `NestedDecorator` | `Nested` | Maps an object or array-of-objects field as an Elasticsearch `nested` type, preserving independence of each object in an array. |
| `FlattenedDecorator` | `Flattened` | Maps an object field as an Elasticsearch `flattened` type. Leaf values are indexed as keywords under a single parent field. Configurable `depthLimit` (default 20) and `index` flag. |
| `ReadOnlyDecorator` | `ReadOnly` | Marks a field as read-only; the value cannot be set or modified by write operations. |

#### Behavior decorators (target: FIELD)

| Decorator class | IDL type string | Description |
|---|---|---|
| `TenantIdDecorator` | `TenantIdDecorator` | Designates the field that stores the tenant identifier. Its presence enables `AdminJsonEntitiesService` and tenant-filtered queries. |
| `VersionDecorator` | `VersionDecorator` | Enables optimistic locking. The version field is automatically incremented on every update; a conflicting update from a stale version fails. |
| `TimeReferenceDecorator` | `TimeReferenceDecorator` | Designates the temporal field for `STREAM`-type entities. Used as the time axis in data stream queries. |
| `DiscriminatorDecorator` | `Discriminator` | Marks a field as the polymorphic type discriminator for union types. Carries an optional `propertyName`. |

#### Access control decorators (target: TYPE, FIELD, FUNCTION)

| Decorator class | IDL type string | Description |
|---|---|---|
| `PolicyDecorator` | `PolicyDecorator` | Defines policy-based access rules as a list of policy lists (`List<List<String>>`). Applied to a type, field, or function via `EntityServiceDecoratorsDecorator`. |
| `RoleDecorator` | `RoleDecorator` | Defines role-based access rules as a list of role names. Applied to a type or function via `EntityServiceDecoratorsDecorator`. |

#### Query decorators (target: FUNCTION)

| Decorator class | IDL type string | Description |
|---|---|---|
| `QueryDecorator` | `Query` | Attaches a SQL statement string to a function definition in the IDL schema. The function becomes a named query executable through `NamedQueriesService`. |

## Usage Example

The following snippet shows how to define an entity schema programmatically, register and publish it, and then perform basic CRUD operations using `EntitiesService`.

```java
// 1. Build the ObjectC3Type schema with decorators
ObjectC3Type personSchema = new ObjectC3Type()
    .setName("Person")
    .setNamespace("com.example.myapp")
    .addProperty("id",        new StringC3Type(), List.of(new AutoGeneratedIdDecorator()))
    .addProperty("firstName", new StringC3Type())
    .addProperty("lastName",  new StringC3Type())
    .addProperty("bio",       new StringC3Type(), List.of(new TextDecorator()))
    .addProperty("notes",     new StringC3Type(), List.of(new NotIndexedDecorator()));

personSchema.addDecorator(new EntityDecorator().setMultiTenancyType(MultiTenancyType.SHARED));

// 2. Create and publish the EntityDefinition
EntityDefinition def = new EntityDefinition()
    .setName("Person")
    .setApplicationId("com.example.myapp")
    .setProjectId("com.example.myapp_default")
    .setSchema(personSchema);

entityDefinitionService.create(def)
    .thenCompose(saved -> entityDefinitionService.publish(saved.getId()))
    .join();

// 3. Write an entity
String defId = "com.example.myapp.Person";
Person p = new Person().setFirstName("Alice").setLastName("Smith");
EntityContext ctx = new DefaultEntityContext(participant);

Person saved = entitiesRepository.save(defId, p, ctx).join();

// 4. Read back a page
Page<Person> page = entitiesRepository.findAll(defId, Pageable.of(0, 20), Person.class, ctx).join();
```

## Notes

- `EntityDefinition` IDs are composed as `applicationId + "." + name` (see `PersistenceUtil.entityDefinitionNameToId`). All service methods that accept an `entityDefinitionId` expect this composite key.
- Index names follow the pattern `{indexPrefix}{slugified-application-id}_{slugified-entity-name}`. The `indexPrefix` defaults to `kinotic_` and is not configurable at the entity level.
- An `EntityDefinition` must be explicitly published before it is visible to the HTTP endpoints or available for data operations. Unpublishing removes it from the served endpoints but does not delete the underlying Elasticsearch index.
- The `update` operation is a partial update (equivalent to Elasticsearch `_update` with a script). Fields absent from the supplied document are left unchanged. `save` is a full replace.
- The `syncIndex` operation calls Elasticsearch's refresh API, making all buffered writes immediately visible to search. It should be used with care in high-throughput scenarios because it bypasses the normal refresh interval.
- `PolicyDecorator` and `RoleDecorator` are `EntityServiceDecorator` subclasses; they are only meaningful when placed inside an `EntityServiceDecoratorsConfig` and attached to a type via `EntityServiceDecoratorsDecorator`. Placing them directly on a field has no authorization effect by itself.
- The `AuthorizationServiceFactory` SPI defaults to a no-op implementation (`NoopAuthorizationServiceFactory`). Applications that require policy or role enforcement must expose a custom `AuthorizationServiceFactory` bean.
- `MultiTenancyType.ISOLATED` and `HYBRID` are present as enum comments but not yet active values. Only `NONE` and `SHARED` are serialized and deserialized.
- Cache invalidation across cluster nodes is handled by Apache Ignite. When an `EntityDefinition` is created, updated, or published, eviction events are broadcast to all nodes so that stale schema and endpoint caches are rebuilt on the next request.
- The number of OpenAPI and GraphQL verticle instances deployed equals `KinoticProperties.maxNumberOfCoresToUse`. The web server verticle (static files and health checks) is deployed as a single instance.
- GraphQL preparsed documents are cached by `CachingPreparsedDocumentProvider` using Caffeine to avoid repeated SDL parsing overhead for repeated queries.
